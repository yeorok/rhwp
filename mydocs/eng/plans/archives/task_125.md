# Task 125 Execution Plan -- Font Metric DB Generation Tool

## Background

### Current Problem
Currently, rhwp's text width measurement calls JS `globalThis.measureTextWidth()` from WASM:

```
Rust layout.rs -> wasm_bindgen -> JS measureTextWidth() -> Canvas measureText()
```

Limitations of this approach:
1. **WASM-JS bridge cost**: JS call per character (thousands per page)
2. **Korean precision deficiency**: All Korean syllables represented by a single 'ga' character width
3. **No offline capability**: Cannot measure without Canvas API (native build, server-side)

### Competitor Analysis Results
Hancom's web viewer embeds glyph widths for 342 fonts across 387 .hft modules, computing text width **server-independently**. Browser measureText is used only as fallback.

### Solution Direction
Create a CLI tool that directly parses the `hmtx` table from TTF/woff2 font files to generate a proprietary font metric DB. Generated metrics are output as Rust source code and embedded in WASM.

### Legal Basis
- Glyph widths (advance width) are functional properties (factual data) of font files and are not subject to copyright protection
- Hancom's .hft files are similarly extracted from the same font properties
- Algorithms (syllable decomposition, etc.) are ideas and not copyrightable
- Data source: 31 woff2 font files already distributed with the project

## Implementation Phases (4 Phases)

---

### Phase 1: TTF Table Parsing Tool

**Goal**: Read `head`, `cmap`, `hmtx`, `maxp` tables from TTF files to extract glyph width data.

**File**: `src/tools/font_metric_gen.rs` (new CLI tool)

**TTF Table Parsing Order**:
1. TTF header: Read table directory (tag/offset/length)
2. `head` table: `unitsPerEm` (em size, offset 18, u16)
3. `maxp` table: `numGlyphs` (glyph count, offset 4, u16)
4. `cmap` table: Unicode -> Glyph ID mapping
   - Format 4: BMP (U+0000~U+FFFF) segment mapping
   - Format 12: Full Unicode (supplementary planes included)
5. `hmtx` table: Glyph ID -> advance width (u16 array)
6. `name` table: Font family name extraction

**Result**: `HashMap<char, u16>` (character -> width in em units)

**Dependencies**: Pure Rust implementation (direct binary parsing without external crates). TTF is big-endian fixed structure, making parsing straightforward.

---

### Phase 2: Korean Syllable Decomposition Compression (type:2 Algorithm)

**Goal**: Compress individual widths of 11,172 Korean syllables (U+AC00~U+D7A3) into initial/medial/final consonant groups.

**Algorithm**:
1. Unicode Korean syllable decomposition: `syllable = (cho * 21 + jung) * 28 + jong`
2. K-means clustering of initial consonants (19) into 2~4 groups by width similarity
3. Clustering of medial vowels (21) into 2~6 groups
4. Clustering of final consonants (28, including none) into 1~3 groups
5. Representative width per group combination = in-group average
6. Error measurement: max/average error between original width and group representative width

**Output**: `HangulMetric { cho_groups, jung_groups, jong_groups, cho_map, jung_map, jong_map, widths }`

---

### Phase 3: Rust Source Code Generation and WASM Embedding

**Goal**: Output generated metric DB as Rust source code to include in WASM at compile time.

**File**: `src/renderer/font_metrics.rs` (generated)

**Generated Code Structure**:
```rust
/// Auto-generated -- created by font_metric_gen tool
pub struct FontMetric {
    pub name: &'static str,
    pub em_size: u16,
    pub latin_widths: &'static [(char, char, &'static [u16])],  // (start, end, widths)
    pub hangul: Option<HangulMetric>,
}

pub struct HangulMetric {
    pub cho_groups: u8,
    pub jung_groups: u8,
    pub jong_groups: u8,
    pub cho_map: &'static [u8; 19],
    pub jung_map: &'static [u8; 21],
    pub jong_map: &'static [u8; 28],
    pub widths: &'static [u16],  // cho_g x jung_g x jong_g
}

pub static FONT_METRICS: &[FontMetric] = &[ ... ];
```

**Lookup Function**:
```rust
pub fn get_char_width(font_name: &str, ch: char) -> Option<u16> {
    let metric = FONT_METRICS.iter().find(|m| m.name == font_name)?;
    if ch >= '\u{AC00}' && ch <= '\u{D7A3}' {
        // Korean syllable decomposition lookup
        metric.hangul.as_ref().map(|h| h.lookup(ch))
    } else {
        // Latin/other range lookup
        metric.latin_widths.iter()
            .find(|(s, e, _)| ch >= *s && ch <= *e)
            .map(|(s, _, w)| w[(ch as u32 - *s as u32) as usize])
    }
}
```

### Phase 4: layout.rs Measurement Pipeline Replacement

**Goal**: Use embedded metrics as primary source in `estimate_text_width()` and `compute_char_positions()`, falling back to JS only for fonts without registered metrics.

**Changed File**: `src/renderer/layout.rs`

**Changes**:
```rust
// [Current]
let raw_px = js_measure_text_width(measure_font, &c.to_string());

// [Changed]
let width = if let Some(w) = font_metrics::get_char_width(font_name, c) {
    // Use embedded metric (em units -> px conversion)
    w as f64 * font_size / metric.em_size as f64
} else {
    // Fallback: JS measureText
    js_measure_text_width(measure_font, &c.to_string()) * font_size / 1000.0
};
```

**Expected Benefits**:
- 90%+ reduction in JS bridge calls (31 registered fonts covered)
- Improved Korean proportional font precision
- Text width measurement possible in native builds
