# Task #30: Excessive Page Count Increase After Paragraph Insert/Delete

## Objective

Fix the bug where paragraph split (Enter) or merge (Backspace) causes an excessive page count increase due to an off-by-one indexing bug in `measure_section_selective`.

## Symptoms

- In `kps-ai.hwp`, executing `splitParagraph(0, 199, 0)` increases page count from 78 → 86 (8-page excessive increase)
- A single line addition causing 8 extra pages is abnormal

## Root Cause Analysis

Index mismatch occurs in `measure_section_selective` after paragraph insertion:

1. `insert_composed_paragraph` executes `bits.insert(para_idx, true)` in `dirty_paragraphs` bitmap → indices shift by 1
2. However, `prev_measured.paragraphs` retains indices from the previous pagination point
3. Result: All non-dirty paragraphs after the insertion point use **cache measurement values shifted by one**
   - Current para 201 (= previous para 200, table 624px) → `prev_measured[201]` (previous para 201, different height)
4. Hundreds of paragraphs get wrong heights applied, distorting pagination results

The same issue exists in `remove_composed_paragraph` (during paragraph merge).

## Implementation Steps

### Step 1: Bug Fix

- `insert_composed_paragraph`: Change to `dirty_paragraphs[section_idx] = None` (force full re-measurement)
- `remove_composed_paragraph`: Same change
- When `dirty_paras = None` in `measure_section_selective`, fall back to `measure_section_incremental` for accurate measurement without index mismatch

### Step 2: Regression Test

- Add `test_split_paragraph_page_count_stability` test
- Load kps-ai.hwp → splitParagraph(0, 199, 0) → verify page count increase <= 2

### Step 3: Existing Test Pass Verification

- Confirm all 778 tests pass
