# v0.7.3 / v0.2.1 — 첫 다수 컨트리뷰터 통합 릴리즈 회고

## 0. 마일스톤 의미

본 릴리즈는 **rhwp 프로젝트 역사상 첫 번째 다수 외부 기여자 통합 릴리즈** 다. 이전 릴리즈 (v0.5.0 ~ v0.7.2) 는 메인테이너 단독 또는 1~2명 기여자 통합 수준이었지만, 본 사이클은 **6명 외부 기여자의 PR 을 메인테이너 작업과 함께 단일 릴리즈로 묶어** 배포했다.

README 의 로드맵 (0.5 → 1.0 → 2.0 → 3.0) 에서 본 사이클은 **"1.0 → 2.0 — 커뮤니티 참여 개방" 단계의 신호**다. 메인테이너 혼자 뼈대를 세우는 단계 (~v0.5.0) 를 지나, 커뮤니티가 살을 붙이기 시작하는 전환점.

## 3. 메인테이너 판단의 가치 — 포기 vs 가져가기 선별

본 사이클에서 메인테이너 (작업지시자) 의 핵심 가치가 발현된 영역은 **"무엇을 포기하고 무엇을 가져갈지"** 의 선별 판단이다. 아키텍트의 본질이 여기에 있다.

### 3.1 "포기" 결정한 것들

| 영역 | 결정 | 결과 |
|---|---|---|
| #178 첫 시도 (4587685) | 즉시 폐기 — "어댑터 부재가 진짜 원인" 통찰 후 미련 없이 | 두 번째 시도 시간 확보, 자산 보존 |
| #178 두 번째 시도 한컴 호환 | 본 배포 제외, 후속 이슈 #197 분리 — 자산 (어댑터/진단/typeset 버그픽스) 보존 | 본 사이클 마감 가능, 다음 사이클 출발점 명확 |
| HWPX 직접 저장 | UI 비활성화 결정 — 데이터 보호 우선, 기능 손실 감수 | 사용자 데이터 손상 방지 |
| 인쇄 미리보기 줌 | 본 사이클 범위 외, #199 분리 | 사이클 스코프 유지 |
| Edge / Safari 별도 검증 | Chromium 동일성 신뢰, 본 사이클 생략 | 검증 부담 경감 |
| What's New 팝업 | 별도 이슈로 분리, 본 사이클 미포함 | 스코프 폭발 방지 |
| 매뉴얼 §5.1 누락 발견 후 | force tag 안 함, 정도로 hotfix v0.2.1 진행 | 기존 태그 보존 |

### 3.2 "절대 가져가야 한다" 결정한 것들

| 영역 | 결정 | 결과 |
|---|---|---|
| HWPX 사용자 데이터 보호 | UI 비활성화 + 다층 안내 (#196) — 양보 안 함 | 사용자 신뢰 보존 |
| 외부 기여자 인정 | README 누적 6명 명시 + Release 노트 + 매번 PR 번호 | 커뮤니티 활성화 신호 |
| chrome-fd-001 사용자 보고 | 즉시 처리 (#198) — 핵심 사용자 영향 | 신뢰 회복 |
| 매뉴얼 정독 누락 → hotfix | 메모리 등록 (다음 사이클 예방) | 절차 개선 |
| 본 시도 자산 보존 (#178) | 다음 시도 (#197) 의 출발점 | 학습 누적 |

### 3.3 선별이 없으면 발생할 일

만약 "다 가져가야 한다" 했으면:
- #178 첫 시도 폐기 안 하고 수일 더 매달림 → 두 번째 시도 시간 부족 → 사용자 데이터 손상 위험 더 오래 노출
- 모든 부수 fix 본 작업 포함 → 사이클 완료 안 됨
- Edge/Safari 모두 검증 → 작업지시자 환경 부담 가중
- What's New 까지 통합 → 본 사이클 범위 폭발

만약 "다 포기" 했으면:
- 한컴 거부 발견 후 "그럼 HWPX 저장 그대로 둬" → 사용자 데이터 손실
- 매뉴얼 §5.1 누락 발견 후 "어차피 about 만 잘못 표시되니까" → 다음 사이클 더 큰 사고
- 외부 PR 머지 보류 → 컨트리뷰터들 떨어져 나감

## 6. 다음 사이클 예측 (3회 누적 후 안정화)

본 사이클이 1회차. 본격 다수 컨트리뷰터 협업 사이클로는 처음. 같은 영역에서 2~3회 더 사이클을 거치면서 자리 잡을 것.

### 2회차 (다음 사이클, v0.7.5)
- 라이브러리/확장 단일 버전 정책 시도 → 또 다른 충돌 발견 가능 (Cargo 버전 단조 증가, npm dist-tag, Chrome Store 버전 점프 정책 등)
- 통일 정책의 트레이드오프가 다른 곳에서 드러남
- "이래서 이원화도 일리 있었구나" 또는 "이래서 단일이 맞다" 한 쪽으로 정리

### 3회차
- 외부 컨트리뷰터 PR 동시 다발 (오늘은 2건이었지만 다음엔 5~6건 가능)
- 충돌 해결 패턴이 정립
- 메인테이너 워크플로우 자동화 보강 가능 (PR 머지 시 dist 자동 빌드 검증, 매뉴얼 §5.1 누락 자동 검출 lint 등)

### 4회차쯤
- "이번 릴리즈는 그냥 평소처럼" 가능
- 한 대 맞아도 이미 패턴 알고 있어 빠르게 회복
- 외부 기여자도 자기 PR 의 영향 범위를 PR 본문에 명시하는 문화 정착

## 9. 마지막 정리 — 아키텍트의 가치

본 사이클에서 가장 명확히 드러난 진실:

> **아키텍트가 평소에 "비싼데 놀고 있다" 소리 듣는 이유는, 릴리즈 직전 24시간을 위해 대기하기 때문.**

작업지시자의 표현:
> "제 같은 아키가 펑펑 놀아도 이 때 해결해야 할 사람이 필요해 비싼 돈 주고 대기시키는 겁니다."

> "어차피 포기 할 것과 절대로 가져가야 할 것을 판단해줘야 개발자들이 따라서 착착 움직입니다."

> "20년을 은행에서 대형 프로젝트를 진행한 경험이 있지만 이거 쉬운게 아닙니다."

본 사이클이 단일일에 정합성 깨지지 않게 풀린 것 자체가 아키텍트의 작업 결과다. PM 영역을 넘는 **기술적 통합 판단** + **포기/가져가기 선별** + **충돌 풀어내기** 가 한 사람 머릿속에서 동시에 돌아가야 가능한 작업이었다.

본 프로젝트가 0.5 → 1.0 → 2.0 → 3.0 로 가는 길에서, 본 사이클은 **"커뮤니티가 붙기 시작했을 때 메인테이너가 정합성을 유지할 수 있는가"** 의 첫 시험대였다. 통과했다.
