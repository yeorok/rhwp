## 1. 배경

### 1.1 사용자 보고 (chrome-fd-001)

> 확장의 기능과 직접적인 연관은 없어보이지만 이 확장을 활성화시키면 브라우저가 마지막 저장 위치를 기억하지 못합니다. 즉 이미지, 웹페이지 등을 "다른 이름으로 저장"할 때 이전 위치가 아닌 바탕화면이 기본값으로 표시됩니다. Windows Chrome 147.0.7727.101.

다운로드 수 10k 도달한 확장의 사용자 직접 보고 — 우선순위 높음.

### 1.2 원인

[`rhwp-chrome/sw/download-interceptor.js:13-22`](rhwp-chrome/sw/download-interceptor.js#L13-L22):

```javascript
chrome.downloads.onDeterminingFilename.addListener((item, suggest) => {
  const filename = item.filename || '';
  if (HWP_EXTENSIONS.test(filename)) {
    handleHwpDownload(item);
  }
  // 기본 파일명 유지 (다운로드는 정상 진행)
  suggest({ filename: item.filename });   // ← HWP 가 아니어도 항상 호출
});
```

Chrome `onDeterminingFilename` API:
- `suggest()` 미호출: Chrome 이 마지막 위치 그대로 사용
- `suggest({ filename })` 호출: Chrome 이 저장 위치 재계산 → 마지막 위치 무력화

→ **HWP 가 아닌 일반 파일 다운로드에도 영향**.

### 1.3 추가 컨텍스트 — 관공서 DEXT5 류 다운로드

작업지시자 통찰: 관공서/공공기관이 흔히 사용하는 DEXT5Upload 같은 다운로드 컨트롤은:
- 토큰 기반 동적 URL (`/download.do?token=xxxxx`)
- POST + Content-Disposition 응답
- 임시 redirect 체인

→ `item.filename` 에 hwp 가 포함되지 않을 수 있음. 현재 인터셉터는 hwp 감지를 놓침. 본 수정에서 함께 개선.

## 2. 목표

### 2.1 1차 목표 (chrome-fd-001 직접 해결)

일반 파일 (이미지/웹페이지/etc) 다운로드 시 Chrome 의 마지막 위치 기억 동작 복원.

### 2.2 2차 목표 (DEXT5 류 hwp 감지율 향상)

`item.filename` 외에 `item.url` / `item.finalUrl` / `item.mime` 도 검사하여 hwp 감지 강화.

### 2.3 성공 기준

- 일반 이미지/PDF/zip 다운로드 → 마지막 저장 위치 기억 정상 복원
- 일반 .hwp/.hwpx 다운로드 (확장자 명시) → 자체 뷰어 정상 트리거 (회귀 0)
- DEXT5 류 hwp (URL/mime 기반) 도 감지율 향상
- chrome-fd-001 보고자 환경 (Windows Chrome 147) 에서 사용자 검증 통과

## 3. 통합 전략 원칙

1. **최소 침습** — 단일 파일 (`download-interceptor.js`) 수정. 다른 sw 모듈 영향 없음
2. **명시 분기** — `if HWP { suggest } else { skip }` 명확화. 모호한 기본값 제거
3. **HWP 감지 다중화** — filename / URL / mime 어느 하나라도 매치되면 hwp 로 처리
4. **회귀 우선** — HWP 인 경우의 동작 (자체 뷰어 트리거) 은 **변경 없음**. 일반 파일 분기만 추가
5. **수동 검증 필수** — chrome-fd-001 보고자 환경 + DEXT5 사이트 모두 작업지시자 검증
6. **트러블슈팅 사전 검색 의무** (메모리 적용) — 본 영역 관련 트러블슈팅 0건 확인됨

## 4. 범위

### 포함

- `rhwp-chrome/sw/download-interceptor.js` 의 `onDeterminingFilename` 리스너 수정
- HWP 감지 다중화 (filename / URL / mime)
- 단위 테스트 추가 (가능 범위)
- 빌드 및 dist 갱신

### 범위 외

- rhwp-chrome 의 다른 인터셉터 (context-menus, message-router, viewer-launcher) 수정
- manifest 권한 변경 (downloads 권한 그대로)
- viewer-launcher 의 다운로드 처리 로직 변경
- Edge / Firefox 확장으로의 포팅
- DEXT5 의 fetch + Blob URL 다운로드 케이스 (별도 이슈로)

## 5. 구현 단계 (3단계 예상)

본 수행계획서는 범위·접근 방향만 확정한다. 구체적 단계별 코드 변경은 구현계획서 (`task_m100_198_impl.md`) 에서 확정.

### Stage 1 — 인터셉터 수정 + 단위 테스트

- `download-interceptor.js`: `if (isHwp) { handleHwpDownload + suggest }` else 분기 명확화
- HWP 감지 다중화: `filename` + `url`/`finalUrl` + `mime`
- 단위 테스트 (가능하면): 인터셉터 동작 분기

완료 기준: 단위 테스트 그린, 빌드 성공.

### Stage 2 — 빌드 + dist 갱신

- `npm run build` 또는 `node build.mjs` 실행
- `rhwp-chrome/dist/` 갱신
- packaging 검증 (manifest, sw 모듈 로드 정상)

완료 기준: 확장 dist 빌드 성공.

### Stage 3 — 작업지시자 수동 검증 + 보고서 + 클로즈

- 작업지시자가 dist 를 Chrome 에 로드 + 검증:
  - 일반 이미지 저장 → 마지막 위치 기억 정상
  - 일반 hwp 다운로드 → 자체 뷰어 정상
  - DEXT5 류 사이트 (관공서) → hwp 감지 동작
- 단계별 보고서 + 최종 보고서
- merge → push → 이슈 close

완료 기준: 작업지시자 수동 검증 통과 + 배포 패키지 준비.

## 6. 위험 요소

| 위험 | 완화 |
|---|---|
| URL/mime 기반 감지가 false positive 발생 | 정규식을 보수적으로 (`\.(hwp\|hwpx)(\?\|$)` 패턴, mime 은 명확한 한컴 mime 만) |
| Chrome 다운로드 API 동작이 OS 별로 미묘하게 다름 | chrome-fd-001 보고자 환경 (Windows Chrome 147) 우선 검증, macOS/Linux 는 후속 |
| DEXT5 의 fetch + Blob 케이스는 본 수정으로 못 잡음 | 본 타스크 범위 외로 명시, 별도 이슈 가능성 기록 |
| dist 빌드 실패 | 빌드 스크립트 (`build.mjs`) 사전 점검 |

## 7. 검증 방법
